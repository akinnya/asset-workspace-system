# 安全策略

本项目欢迎负责任的安全报告。若你发现可能影响仓库使用者、部署环境或数据边界的安全问题，请按照以下流程处理。

## 报告范围

优先报告以下问题：

- 身份认证、权限控制、越权访问
- 文件上传、文件读取、路径处理
- SQL 注入、命令执行、模板注入、反序列化
- 敏感日志泄露、配置泄露、令牌泄露
- 公开链接、分享能力、工作区协作链路中的访问控制缺陷

以下情况通常不属于本仓库代码漏洞：

- 你自己的部署环境使用弱口令、默认网络暴露或错误的反向代理配置
- 第三方平台自身的服务故障或条款限制
- 未修改代码前提下，完全由本地开发配置导致的非默认暴露

## 报告方式

1. 优先使用 GitHub 的私密漏洞报告能力。
2. 如果仓库尚未开启私密报告，请不要在公开 issue 中直接披露利用细节。
3. 你可以先提交一个不含攻击细节的说明，请维护者建立私密沟通渠道。

## 报告内容建议

- 受影响模块或接口
- 影响范围与安全后果
- 复现前提、触发步骤与边界条件
- 是否需要身份、特定角色或特定配置
- 修复建议或临时缓解措施

请不要在公开渠道中直接粘贴以下内容：

- 真实访问令牌、数据库凭据、邮件凭据或云服务密钥
- 含个人信息的数据样本
- 可直接复用的破坏性脚本或完整攻击载荷

## 处理目标

- 维护者目标是在收到报告后尽快确认问题是否成立。
- 对可复现且影响明确的问题，优先给出修复路线或缓解建议。
- 在补丁发布前，避免公开披露足以直接利用的细节。

## 发布原则

安全修复合并后，可在确认风险已被控制的前提下公开披露问题背景、影响范围和修复方式。
